# 星位置合わせ改善ロードマップ

## 現状

現在の位置合わせ処理は以下の流れになっている。

1. プレビュー画像生成
2. 特徴点検出または星検出
3. 対応点生成
4. アフィン変換行列推定
5. クライアントサイドで本画像へ適用
6. 合成

星検出アルゴリズムは実装済み。以前より周辺部の星流れは改善したが、画像周辺ではまだ星が線状になるケースがある。

## 直近で対応すること

### 検出手法の設定タイミング変更

現在はプレビュー結果画面で `特徴点検出` / `星検出` を切り替えている。しかし、位置合わせ方式はプレビュー生成と位置合わせ推定を開始する前に選ぶべき設定である。

対応:

- 検出手法の選択を、プレビュー生成前の画面へ移動する。
- 初心者向けの通常 UI では専門用語を抑え、将来的には `標準` / `高精度` のような表現を検討する。
- 開発中は `アフィン` / `ホモグラフィ` などの方式名を staging debug または開発向け表示で扱ってよい。

## 位置合わせ手法ロードマップ

### Phase 1: 星検出 + アフィン

現在の基本方式。特徴点検出または星検出で対応点を作り、RANSAC でアフィン変換を推定し、画像全体へ適用して合成する。

流れ:

1. 特徴点検出または星検出
2. 対応点生成
3. RANSAC
4. アフィン推定
5. 画像全体へ適用
6. 合成

表現できる変換:

- 平行移動
- 回転
- 拡大縮小
- せん断

実装状況: 実装済み。

### Phase 1.5: ホモグラフィ追加

アフィン変換の上位互換となる射影変換を追加する。

目的:

- アフィンとの比較
- 周辺部がどの程度改善するか評価
- 星景写真への効果確認
- 将来の局所ワープ前処理候補

流れ:

1. 特徴点検出または星検出
2. 対応点生成
3. RANSAC
4. ホモグラフィ推定
5. 画像全体へ適用
6. 合成

評価項目:

- 平均残差
- 中央値残差
- RANSAC inlier 率
- 周辺部の星流れ
- 合成品質

ホモグラフィは固定三脚の星空では根本解決にはならない可能性が高い。それでも実装コストが比較的低いため、比較対象および局所ワープ前処理候補として追加する。

### Phase 2: 局所ワープ

このフェーズでは星検出自体を改善するのではなく、既に得られている対応星情報をさらに有効利用する。

アフィンまたはホモグラフィ適用後、各対応星には予測位置と実際の位置の差、つまり残差が残る。この残差を画像全体へ補間し、局所的な補正を行う。

流れ:

1. 星検出
2. 対応点生成
3. アフィンまたはホモグラフィ
4. 各対応星の残差計算
5. 残差ベクトル生成
6. 画像全体へ補間
7. 局所ワープ
8. 合成

このフェーズで初めて追加するもの:

- メッシュワープ
- Thin Plate Spline (TPS)
- 多項式ワープ

画像を格子状に分割したり、残差を補間したりする処理はすべてこのフェーズで扱う。

#### メッシュワープ

画像を格子状に分割し、各格子点の補正量を周辺の星残差から推定して、画像全体へ滑らかに補間する。

メリット:

- 実装難易度が比較的低い
- 局所補正できる
- 周辺部改善が期待できる

#### Thin Plate Spline

対応星を制御点として、画像全体をゴム板のように滑らかに変形する。

メリット:

- 境界が自然
- メッシュより滑らか
- 星対応点を直接利用できる

注意:

- 外れ値に弱い
- 対応点品質が重要

#### 多項式ワープ

座標変換を多項式で近似する。レンズ歪み補正にも利用できる。

例:

```text
x' = a0 + a1x + a2y + a3xy + ...
```

### Phase 3: 天球モデル

最終形。画像を直接合わせるのではなく、撮影時のカメラ姿勢を推定する。

流れ:

1. 星検出
2. プレートソルブ
3. 撮影方向推定
4. 天球上で回転
5. 画像へ再投影
6. 合成

メリット:

- 理論的に最も正確
- 周辺部まで点像化しやすい
- レンズ歪みも扱える

デメリット:

- 実装難易度が非常に高い
- 星表照合が必要
- 計算量が多い

## 実装優先順位

1. 検出手法選択をプレビュー前へ移動する。
2. アフィン変換を安定化する。
3. ホモグラフィを実装する。
4. アフィンとホモグラフィを比較評価する。
5. 対応星残差を可視化する。
6. メッシュワープを実装する。
7. TPS を検証する。
8. 多項式ワープを検証する。
9. 天球モデルを検討する。

## UI 方針

初心者向けには専門用語を隠す。

通常 UI の例:

- 位置合わせ: 標準
- 位置合わせ: 高精度

内部対応の例:

- 標準: アフィン
- 高精度: ホモグラフィ、将来は局所ワープ

開発中は、位置合わせ方式として `アフィン` / `ホモグラフィ` を直接表示してもよい。

## 将来的な内部アーキテクチャ

変換方式をインターフェース化する。

```text
AlignmentEstimator
├── AffineEstimator
├── HomographyEstimator
├── MeshWarpEstimator
├── TPSEstimator
└── SphericalEstimator
```

局所ワープでは以下の構成にする。

```text
Global Transform
↓
Residual Field
↓
Local Warp
```

`Global Transform` はアフィンとホモグラフィを切り替えられる設計にする。

## 最初の実装プラン

最初の実装は、UI の設定タイミング変更とホモグラフィを追加するための設計準備に絞る。

### 1. 検出手法選択をプレビュー前へ移動

- 現在の `stars` / `akaze` 選択 UI を、プレビュー生成・アップロード・位置合わせ開始より前の設定領域へ移す。
- プレビュー結果画面では、選択済み方式の表示に留める。
- 方式を変更した場合は既存の preview alignment 結果を再利用せず、再実行が必要な状態にする。
- API の `alignmentMethod` は既存の `stars` / `akaze` を維持する。

### 2. グローバル変換方式の概念を追加

- 現在の検出手法 `stars` / `akaze` と、変換モデル `affine` / `homography` を分けて扱う設計にする。
- 直近の API 互換を優先し、最初は内部型だけを整理する。
- Protocol Buffers の拡張は、UI と worker の最小実装方針が固まってから行う。

### 3. ホモグラフィ実装の下準備

- `crates/stellacomp` に affine 推定処理と homography 推定処理を並べられる境界を作る。
- 対応点生成までは既存の `stars` / `akaze` を共用する。
- `estimateAffinePartial2D` と同じ入力点列から `findHomography` を呼べるようにする。
- 成功時は 3x3 行列、失敗時は既存と同様に warning + fallback できる方針にする。

### 4. 比較評価の診断値を追加

- 平均残差
- 中央値残差
- RANSAC inlier 率
- 画像中心部と周辺部の残差
- 合成結果の目視比較メモ

これらは通常 UI に常時表示しない。CLI example または staging debug で確認できるようにする。

### 5. 対応星残差可視化の準備

- Phase 2 の前段として、アフィンまたはホモグラフィ適用後の対応星残差を取得できるようにする。
- 最初は画像への overlay 実装までは入れず、CLI または JSON/TSV 出力で確認する。

## 採用基準

- 既存の preview alignment API と `stars` / `akaze` の互換性を壊さない。
- `stars + affine` の現在挙動を標準として維持する。
- ホモグラフィは最初から標準化せず、比較評価できる実験的方式として追加する。
- 周辺部改善を評価するときは、平均残差だけでなく中央値残差、inlier 率、周辺部の星流れ、合成品質を合わせて見る。
- 局所ワープ、TPS、多項式ワープは Phase 2 まで実装しない。
- 天球モデルは Phase 3 の検討対象とし、当面の実装には含めない。

## まとめ

現在は星検出とアフィンまで実装済み。次はホモグラフィを追加して評価する。その後、対応星残差の計算から局所ワープへ進み、最終的には天球モデルを目指す。

このロードマップにより、実装難易度を段階的に上げながら、画質改善の効果を確認していく。
